# Project Overview

Last verified against repo state: 2026-05-02, `HEAD` commit `53788d2`.

This repository is an EV smart-charging monorepo for a Dundee-focused recommendation and simulator workflow. The current user-facing goal is: a mobile app submits a charging request JSON, the FastAPI backend injects it into the standalone runtime, Dundee charging stations are ranked, and recommendation options are returned to the mobile app while runtime state is visible in a dashboard.

The long-term goal is smart charging recommendations: first with deterministic baselines and stronger tests, later with learned policy inference from MARL checkpoints. The current implementation is not MARL/RL inference. Station recommendations are generated by deterministic heuristic scoring in `packages/ev_core/src/ev_core/recommender/ranker.py`.

## Architecture

- `apps/mobile`: Expo React Native app. It collects target SOC, preference mode, and charger type, then calls FastAPI `POST /recommendations`.
- `apps/api`: FastAPI gateway. The recommendation endpoint is thin and delegates to runtime services. Station CRUD still uses mock in-memory station data.
- `services/sim_runtime`: standalone runtime manager and persistence layer around `ev_core.env.DundeeEnv`.
- `packages/ev_core`: reusable Python core containing request/response contracts, Dundee simulator environment, runtime entities, baseline allocation policies, recommender ranker/service, data repository loaders, forecasting interfaces, and time utilities.
- `dashboards/sim_dashboard`: Streamlit dashboard. It reads runtime storage directly and can control the simulator runtime.
- `data`: raw/interim/processed Dundee data artifacts used by the simulator.
- `outputs`: generated QC reports, figures, maps, and live runtime persistence.
- `scripts`: data-building scripts and runtime helper scripts.
- Root deployment files: `Dockerfile.api`, `Dockerfile.dashboard`, `Dockerfile.mobile`, `docker-compose.yml`, `requirements.txt`, and `.dockerignore` support containerized runtime/API/dashboard/mobile deployment.

## Current Development Target

The immediate development target should be to make recommendation infrastructure clean and testable while preserving current mobile/API/dashboard response shape. A good target interface is a policy-level `rank(request, candidates, runtime_context)` abstraction, with the existing `WeightedHeuristicRanker` wrapped as a baseline weighted-score policy before MARL checkpoint inference is introduced.

## Current Truth

- FastAPI `/recommendations` exists in `apps/api/app/routers/recommendations.py`.
- `apps/api` imports `ev_core` contracts and runtime manager code through bootstrap paths.
- Mobile recommendation calls are wired in `apps/mobile/src/services/api.ts`; the API base URL now uses `process.env.EXPO_PUBLIC_API_BASE_URL` when set, otherwise falls back to local Expo/Android defaults.
- API CORS origins are configurable through `CORS_ORIGINS` in `apps/api/app/main.py`, defaulting to local Expo/web origins.
- Runtime state and recommendation history are persisted under `outputs/runtime`.
- Dashboard reads runtime storage directly through `services.sim_runtime.storage.RuntimeStorage`.
- Current recommendation scoring is deterministic weighted heuristic scoring, not random, mock, RL, or MARL.
- Runtime allocation baselines exist separately in `packages/ev_core/src/ev_core/env/baselines.py`; they choose among already-ranked recommendation options during simulation steps.
